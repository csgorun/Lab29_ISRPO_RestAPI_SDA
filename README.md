## 🎯 Краткое описание работы

В ходе данной лабораторной работы были изучены и применены на практике следующие концепции:

- **Принципы REST-архитектуры**: клиент-сервер, stateless, единый интерфейс, кэшируемость, слоистость.
- **Создание Controller-based API** в ASP.NET Core с полным набором CRUD-операций.
- **Разница между Minimal API и контроллерами**: когда и какой подход использовать.

- **Тестирование API** через Swagger UI и расширение REST Client в VS Code.
- **Использование DTO** для валидации входящих данных и защиты от некорректных запросов.
- **Настройка CORS** для будущего взаимодействия с фронтендом.

## 🗂️ Реализованные маршруты API

| Метод | Маршрут | Описание | Параметры | Статусы |
|-------|---------|----------|-----------|---------|
| `GET` | `/api/tasks` | Получить все задачи | `?completed=true/false` (query) | `200 OK` |
| `GET` | `/api/tasks/{id}` | Получить задачу по ID | `id` (path) | `200 OK`, `404 Not Found` |
| `POST` | `/api/tasks` | Создать новую задачу | `CreateTaskDto` (body) | `201 Created`, `400 Bad Request` |
| `PUT` | `/api/tasks/{id}` | Полностью обновить задачу | `id` (path), `UpdateTaskDto` (body) | `200 OK`, `404 Not Found` |
| `PATCH` | `/api/tasks/{id}/toggle` | Переключить статус выполнения | `id` (path) | `200 OK`, `404 Not Found` |
| `DELETE` | `/api/tasks/{id}` | Удалить задачу | `id` (path) | `204 No Content`, `404 Not Found` |
| `GET` | `/api/tasks/search` | Поиск задач по заголовку/описанию | `?query=...` (query) | `200 OK` |
| `GET` | `/api/tasks/sorted` | Получить задачи с сортировкой | `?sortBy=priority/title/createdAt` (query) | `200 OK` |
| `GET` | `/api/tasks/priority/{level}` | Фильтрация по приоритету | `level` (path: Low/Normal/High) | `200 OK` |
| `GET` | `/api/tasks/stats` | Статистика по задачам | — | `200 OK` |

## ⚙️ Итоговая таблица: ASP.NET Core Controller-based API

| Аспект | Реализация в ASP.NET Core |
|--------|---------------------------|
| **Маршруты** | Группировка через атрибут `[Route("api/[controller]")]` |
| **Базовый URL** | Формируется автоматически: `[controller]` → имя класса без `Controller` |
| **Параметр пути** | `(int id)` в сигнатуре метода + атрибут `[HttpGet("{id}")]` |
| **Параметр запроса** | Атрибут `[FromQuery]`, например: `[FromQuery] bool? completed` |
| **Тело запроса** | Атрибут `[FromBody]` + DTO-класс, например: `[FromBody] CreateTaskDto dto` |
| **Ответ 200** | `return Ok(data)` |
| **Ответ 201** | `return CreatedAtAction(nameof(GetById), new { id = newTask.Id }, newTask)` |
| **Ответ 404** | `return NotFound("Задача не найдена")` |
| **Ответ 400** | `return BadRequest("Заголовок обязателен")` |
| **Ответ 204** | `return NoContent()` |
| **Типизация данных** | Строгая статическая типизация C# + модели и DTO |
| **Документация** | Swagger UI — устанавливается через пакет `Swashbuckle.AspNetCore` |

## ✅ Главные выводы

1. **REST — это архитектурный стиль, а не протокол.**  
   Те же принципы (ресурсы, HTTP-методы, статусы) применяются независимо от языка: в JavaScript (Express), Python (FastAPI), Java (Spring) или C# (ASP.NET Core).

2. **Контроллеры обеспечивают чистую архитектуру.**  
   В отличие от Minimal API, контроллеры позволяют логически группировать маршруты, выносить общую логику и масштабировать проект без хаоса в `Program.cs`.

3. **DTO защищают API от некорректных данных.**  
   Клиент не может передать служебные поля (`Id`, `CreatedAt`), а сервер сам устанавливает значения по умолчанию — это снижает риск ошибок и уязвимостей.

4. **Правильные HTTP-статусы — часть «контракта» API.**  
   Клиент (например, фронтенд на JS) по коду ответа понимает, что произошло: `201` — ресурс создан, `404` — не найден, `400` — ошибка в данных. Это делает взаимодействие предсказуемым.

5. **Swagger UI и REST Client экономят время на тестировании.**  
   Не нужно писать отдельный тестовый клиент: документация генерируется автоматически, а запросы можно хранить в репозитории в файле `.http`.
